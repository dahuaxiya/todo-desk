# Todo Desk 仓库协作规则

## 功能改动必须使用独立分支

- 当用户已经确认要新增功能，或要修改、修复现有功能行为时，开始修改代码前必须确认当前工作位于该任务的独立分支。
- 每次新建任务分支时，无论当前检出的是 `main` 还是其他功能分支，都必须显式以本地 `main` 指针为起点，例如 `git switch -c feature/<功能名> main` 或 `git switch -c fix/<问题名> main`。不得直接执行不带起点的 `git switch -c <新分支>`、`git checkout -b <新分支>`，避免从当前功能分支继续分叉。
- 新建后、产生任何新提交前，必须确认新分支的 `HEAD` 与当前 `main` 完全一致，例如执行 `test "$(git rev-parse HEAD)" = "$(git rev-parse main)"`。校验失败时停止开发，先纠正分支起点。
- 当前工作区有未提交改动，或 `main` 已被其他 worktree 检出时，不得为创建新分支而移动、暂存、覆盖或携带这些改动。应使用 `git worktree add -b <新分支> <新目录> main` 从 `main` 创建独立工作区；除非用户明确同意，不得擅自 stash 或提交现有改动。
- 如果当前已经位于与本次任务对应的独立分支，继续使用该分支，不要为同一个任务重复创建嵌套分支。
- 不要因为阅读代码、讨论方案、回答问题或仅修改说明文档而机械创建功能分支；只有用户确认进入功能实现或旧功能改动阶段后才触发该规则。

## 测试新能力不得关闭已有 Todo Desk

- 测试 Todo Desk 新能力前，必须先记录当前正在运行的 Todo Desk PID、API 端口和健康状态。不得退出、重启、替换或结束已有 Todo Desk 进程，也不得使用 `killall`、`pkill` 或按进程名称批量结束实例。
- 必须额外启动一个隔离的测试进程。测试实例必须同时使用独立的 `--user-data-dir` 和未被占用的 API 端口，禁止与现有实例共用生产数据目录或监听端口。需要真实任务数据时，只能把生产数据文件复制到临时目录后修改副本中的 `settings.apiPort`，不得直接修改生产数据。
- 测试源码时优先直接启动当前仓库的 Electron，例如 `./node_modules/.bin/electron . --user-data-dir="$TEST_USER_DATA"`；测试已安装版本时使用 `/Applications/Todo Desk.app/Contents/MacOS/Todo Desk --user-data-dir="$TEST_USER_DATA"`。启动前必须在临时数据文件中写入测试端口，并确认该端口空闲。
- 启动测试实例后，必须同时验证已有实例和测试实例的 `/health`，确认两者 PID、数据目录和 API 端口互不相同，再开始测试新能力。
- 对 UI、交互或其他需要用户亲自查看的改动，隔离测试实例在 Agent 完成自动验证后必须继续保持运行。交付时应告诉用户测试窗口、PID、API 端口和必要的使用入口，不得在同一轮回复前停止进程或删除测试数据。
- 只有用户明确表示已经体验完成、允许关闭或要求清理后，才能停止为用户保留的测试实例；必须只停止此前记录的精确 PID，然后删除对应临时目录，并再次验证原有 Todo Desk 进程和健康接口仍正常。
- 纯 Agent 内部的自动化测试实例如果不需要交给用户查看，可以在验证结束后停止并清理。无法建立隔离测试实例时，应报告测试阻塞，不得关闭已有实例绕过问题。

## 完成后必须询问是否合并

- 当 agent 认为实现、必要测试和验证均已完成时，必须先向用户说明当前分支、主要改动和验证结果，并明确询问是否将该分支合并回 `main`。
- 在用户明确同意前，不得自行切换到 `main` 执行合并，也不得以“任务已完成”为理由自动合并。
- 用户同意后，合并前仍需检查工作区、提交状态和潜在冲突；发现风险时先说明情况，不得覆盖用户已有改动。
